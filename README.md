# QA API Lab (FastAPI) — портфолио-проект для тестирования API

Этот репозиторий показывает базовые навыки QA по API: запуск сервера, ручное тестирование через Swagger/Postman и автопроверки (pytest) с CI в GitHub Actions.

## Что внутри

API endpoints:
- `GET /health` — проверка, что сервис жив
- `POST /login` — логин по email/password (демо-логика)

## Как запустить локально (Windows)

Это небольшой учебный проект, который я сделал как тренировку для QA/Manual в сторону API-тестирования. Здесь есть простой эндпоинт `/login`, который показывает базовую “правильную” логику: сервер сам проверяет входные данные и отдаёт корректные HTTP статус-коды, а тесты это фиксируют и проверяют в CI.

Проект специально сделан максимально понятным: можно открыть страницу, ввести данные и нажать кнопку, а можно тестировать напрямую через API (как это и делает CI). Важно понимать, что интерфейс на странице помогает пользователю, но не защищает сервер — поэтому все реальные проверки находятся на стороне API.

## Что делает `/login`

Сервер принимает JSON с `email` и `password` и ведёт себя так:

Если пароль не проходит политику (длина, наличие заглавной/строчной буквы и цифры, только буквы/цифры), сервер возвращает `422 Unprocessable Entity`. Это означает “данные не прошли валидацию”.

Если формат корректный, но логин/пароль неправильные, сервер возвращает `401 Unauthorized`. Это означает “попытка входа есть, но данные не совпали”.

Если всё корректно и совпало с тестовыми кредами, сервер возвращает `200 OK` и JSON со статусом “ok”.

## Почему это полезно для QA

В этом проекте на практике видно разницу между проблемой формата (422) и проблемой авторизации (401). Тесты проверяют поведение API напрямую, без фронта, как в настоящих интеграционных проверках. Плюс настроен GitHub Actions, чтобы при любом пуше тесты прогонялись автоматически и сразу было видно, что сломалось.

## Технологии

FastAPI используется как каркас API. Pydantic валидирует входные данные. Pytest прогоняет автотесты. GitHub Actions запускает тесты в CI.
